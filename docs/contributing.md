# Contributing

Thanks for considering a contribution. The project follows the standard
[Kubebuilder](https://book.kubebuilder.io/) layout and the standard
Kubernetes operator workflow — if you've worked on a controller before,
nothing here will surprise you.

---

## Set up the dev environment

### Prerequisites

- Go **1.25.3** or later (`go version` to check).
- Docker for building images and running the e2e test cluster.
- `kubectl`, `kustomize`, `helm` v3.8+.
- `kind` for the e2e test cluster.

The Makefile pulls the rest (`controller-gen`, `setup-envtest`,
`golangci-lint`, `kustomize`) into `./bin` on first use.

### Clone and bootstrap

```bash
git clone https://github.com/BEIRDINH0S/sonarqube-operator
cd sonarqube-operator
go mod tidy

# This downloads controller-gen, setup-envtest, etc. into ./bin
make manifests generate fmt vet
```

You should end up with no diff:

```bash
git status
# nothing to commit, working tree clean
```

If `make manifests` produces a diff, **commit it** — the generated CRDs
in `config/crd/bases/` and RBAC in `config/rbac/role.yaml` must always
match the markers in `api/v1alpha1/*_types.go`.

## Run the test suite

### Unit + envtest

```bash
make test
```

This runs `go test ./...` against an in-memory Kubernetes API
([envtest](https://book.kubebuilder.io/reference/envtest.html)). Fast
(~30s) and the right loop for iterating on controllers.

### End-to-end on kind

```bash
make test-e2e
```

This:

1. Spins up a fresh kind cluster (`sonarqube-operator-test-e2e`).
2. Installs the operator from a locally-built image.
3. Runs the full reconcile flow against a real SonarQube + PostgreSQL.

Takes ~5–8 minutes. Used by CI on every PR.

### Lint

```bash
make lint        # check
make lint-fix    # auto-fix what's safe
```

`golangci-lint` runs with the project's `.golangci.yml`. The CI rejects
PRs with lint warnings — keep it clean locally.

## Project layout

The standard Kubebuilder layout. The pieces you're likely to touch:

| Path | What's there |
|---|---|
| `api/v1alpha1/*_types.go` | CRD schemas. Edit these to add fields or validation, then run `make manifests generate`. |
| `internal/controller/*` | Reconciliation logic. One file per CRD, plus shared helpers. |
| `internal/sonarqube/` | The typed REST client for SonarQube. |
| `internal/metrics/` | Prometheus metric definitions. |
| `config/samples/` | Example CRs (one per CRD). |
| `test/e2e/` | End-to-end test suite. |
| `docs/` | This MkDocs site. |
| `charts/sonarqube-operator/` | The Helm chart. |

**Do not edit:**

- `config/crd/bases/*.yaml` — generated by `make manifests`.
- `config/rbac/role.yaml` — generated by `make manifests`.
- `api/v1alpha1/zz_generated.*.go` — generated by `make generate`.
- `PROJECT` — managed by `kubebuilder`.

## After making changes

The reflexes:

```bash
# After editing api/v1alpha1/*_types.go or any +kubebuilder marker:
make manifests
make generate

# After editing any *.go:
make fmt
make vet
make lint
make test

# After editing the Helm chart:
make helm-sync-crds   # if you changed CRDs
make helm-lint
```

The full sanity sweep before pushing:

```bash
make manifests generate fmt vet lint test
```

## Commit message style

Use [Conventional Commits](https://www.conventionalcommits.org/) prefixes.
A few examples from the project history:

```
feat: phase 5 — leader election + rate limiting
fix: token rotation now triggers when secret is deleted
test: add envtest coverage for plugin restart batching
docs: phase 7.2 — fill Getting Started section with real content
```

The first line is a one-sentence summary in the imperative mood. The
body (after a blank line) explains *why* — the *what* is in the diff.

**Do not include AI / Claude / Copilot mentions in commit messages.** No
`Co-Authored-By: Claude` trailer, no `🤖 Generated with` footer. The
project history reads as written by a human author.

## Pull request workflow

1. **Branch off `main`** with a descriptive name:
   `feat/quality-gate-import`, `fix/token-rotation-race`, etc.
2. **Make focused commits** — one logical change per commit. If you find
   yourself bundling unrelated fixes, split them.
3. **Write the PR description** with:
   - The problem the PR solves
   - The approach taken (with rationale for non-obvious choices)
   - How you tested it
4. **CI must pass** — `lint`, `test`, `test-e2e` all green before review.
5. **Address review comments** in **fixup commits**, then rebase before
   merge. Don't squash everything into one giant commit.

## Adding a new CRD field

Worked example — let's add `spec.timezone` to `SonarQubeInstance`.

1. Edit `api/v1alpha1/sonarqubeinstance_types.go`:
   ```go
   // timezone configures the SonarQube server's timezone (e.g. "Europe/Paris").
   // +optional
   Timezone string `json:"timezone,omitempty"`
   ```
2. Regenerate:
   ```bash
   make manifests generate
   ```
3. Use the field in `internal/controller/sonarqubeinstance_controller.go`,
   typically by adding an env var to the StatefulSet pod template:
   ```go
   if instance.Spec.Timezone != "" {
       envVars = append(envVars, corev1.EnvVar{
           Name: "TZ", Value: instance.Spec.Timezone,
       })
   }
   ```
4. Add a unit test in `internal/controller/sonarqubeinstance_controller_test.go`.
5. Update the [reference doc](reference/crds/sonarqubeinstance.md) with
   the new field.
6. Update one of the samples in `config/samples/` if relevant.
7. Run the full sweep, push, open a PR.

## Releasing

Tagged releases trigger the [Release workflow](https://github.com/BEIRDINH0S/sonarqube-operator/actions/workflows/release.yml).
The flow:

```bash
git tag v0.6.0
git push origin v0.6.0
```

Within ~10 minutes the workflow:

1. Runs the full validation suite (lint + test).
2. Builds and pushes a multi-arch image (amd64 + arm64) with three tags
   on stable releases (`vX.Y.Z`, `vX.Y`, `latest`) and one tag on
   pre-releases (`vX.Y.Z-rc.N`).
3. Generates `dist/install.yaml`.
4. Packages the Helm chart and pushes it to GHCR OCI.
5. Creates a GitHub Release with `install.yaml` and the chart `.tgz`
   attached, marked as pre-release if the tag has a `-`.

Pre-release a candidate first (`v0.6.0-rc.1`), test it, then tag the
final.
